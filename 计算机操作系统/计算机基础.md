### 新能测试常见术语

```
1. 负载
	对被测试系统不断加压力，直到新能指标超过预期或某项资源达到饱和，以验证系统处理极限，为系统性能调优提供数据
2.并发
	1.狭义上的并发：所有用户在同一时间点进行同样的操作，一般指同一类型的场景
	2.广义上的并发：多个用户与系统发生交互，这些业务场景可以是相同的也可以是不同的，交叉请求处理比较多
3.压力
	系统在一定饱和状态下，
	特点：主要目的是检查系统处于压力情况下应用性能表现，重点在于有无出错信息产生，系统对应用的响应时间等
4.事务
	事务是从端到端，一个完整的操作过程，
5.吞吐量





```

### 安开发阶段划分
```
单元测试（白盒测试为主）
	单元测试又称模块测测试，是针对软件设计最小单元————程序模块进行正确性检查测试工作。其目的在一检查每个程序单元能否正确实现详细设计说明种的模块功能，性能，接口和设计约束等要求，发现各个模块内部可能存在的各种错误。 单元测试需要从程序的内部结构出发设计测试用例。多个模块可以平行独立进行单元测试
集成测试 
	集成测试也叫做组装测试。通常在单元测试的基础上，将所有程序模块进行有序，递增的测试，基础测试是检验程序单元或部件的接口关系，逐步集成为符合概要设计要求的程序部件或整个系统
确认测试 （黑盒测试为主，）
	确认测试也叫有效测试，是在模拟的环境下，验证软件所有功能和性能以及其他特性是否与用户的预期要求一直。通过了确认测试之后的软件，才具备进入系统测试阶段的资格
系统测试 （黑盒测试为）
	系统测试是真实的系统运行环境下，检查完整的程序系统是否和系统（包括硬件，外设，网络和系统软件，支持平台等）正确配置，连接，并最终满足用户所有需求
验收测试
	是软件产品验证的最后一个环节，按照项目任务书或合同，供需双方约定的验证依据文档进行的对整个系统的测试与评审。决定是否接受或拒收系统
	ab测试
	阿尔法测试， ——————>由 公司内部人员（除去开发）把自己当作成用户完成的验收测试(内测)
	β测试  ————>由 实际用户完成的验收测试（公测）

```

### 安测试技术划分

```

黑盒测试   （动态为主，静态为辅）
	通过软件的外部表现来其缺陷和错误，黑盒测试把测试对象看成一个黑盒子，完全不考虑程序内部结构和处理过程。黑盒测试是在程序界面进行测试，它只是检查样序是否按照需求规定说明书的规定正常实现。  （只关注输入输出）
	黑盒测试大部分是功能测试， 但是黑盒测试不是功能测试

白盒测试 （静态为主，动态为辅）
	通过对程序内部结构的分析，检查来寻找问题，白盒测试可以把程序看成装在一个透明的白盒里，也就是清楚了解程序结构和处理过程，检查是否所有的结构及路径是正确的，检查软件内部动作是否按照设计说明的规定正常进行，白盒测试又称结构测试 （关注代码内部逻辑结构,路径是否正确）

灰盒测试  （外部测试，网站 B/S结构的）
	介于白盒与黑盒测试之间的测试，灰盒测试关注输出对于输入的正确性；同时也关注内部表现，但这种关注不像白盒测试那样详细，完整，只是通过一些表征的现象，事件，标准来判断内部的运行状态。
	灰盒测试结合了白盒测试和黑盒测试要素，它考虑了用户端，特定的系统只是和操作环节。它在系统组件的协同性能环境中评价应用软件的设计

```

### 安代码运行划分

```
静态测试
	指不实际运行被测试对象，而只是静态地检查程序代码，界面或文档中可能存在错误的过程
	代码测试：主要测试代码是否符合相应的标准，规范
	界面测试：主要测试软件的实际界面与需求中的说明是否符合
	文档测试：主要测试用户手册和需求是否真正符合用户的实际需求
	
动态测试
	制实际运行被测对象，输入相应的测试数据，测试实际输入结果和预期结果是否相符的过程。所以我们判断一个测试属于动态测试还是静态测试，唯一的标准就是看是否运行程序

安软件特性分类

	1. 功能测试

	2.性能测试
		功能的另一个指标，主要关注软件中某一功能在指定的时间，空间条件下，是否使用正常
		软件性能包括很多方面，主要有时间性能和空间性能2种

其他分类
	
	1. 回归测试
		是指对软件的新版测试时，重复执行之前某一个重要版本的所有测试用例
		目的：
			验证之前版本产生的所以缺陷已全部被修复
			确认修复这些缺陷没有应发新的缺陷

	2.冒烟测试 （项目开始）
		是指在对一个新版本进行系统大规模的测试之前，先验证一下软件的基本功能是否实现，是否具备可测性，所以也叫可测性测试
		对软件的基本功能做最初级的验证， 如果冒烟测试没有通过，后续测试不予展开

	3.随机测试
		也称为随意测试，是指测试人员基于经验和直觉的探索性测试，其目的是模拟用户的真实操作，并发现一些边缘性错误
```

### 软件测试原则

```(^_^) (^_^)

1.	所以测试的标准都是建立在用户需求上。
2.	软件测试必须基于‘质量第一’ 的思想去开展各项工作，但时间和质量冲突时，时间要服从质量
3.	事先定义好好产品的质量标准，只要有有了质量标准，才能根据测试的结果，对于产品的质量进行分析和评估
4.	软件项目一启动，软件测试也就是开始，而不是等程序写完，才开始进行测试。
5.	穷举测试是不可能的。甚至一个大小适合的程度，其路径排列的数量也非常哒，因此，在测试中不可能运行路径的每一种组合
6.	第三方进行测试会更客观，更有效。
7.	软件测试计划是做好软件测试工作的前程
8.	测试用例是设计出来的，不是写出来的，所以要根据测试的目的，采用相应的方法设计测试用例,从而提高测试的效率，更多地发现错误，提高程序的可靠性。
9.	对发现错误较多的程序段，应该进行更加深入的测试。一段程序中已发现的错误数越多，其中存在的错误概率也就越大
10.	重视文档，妥善保持一切测试过程文档（测试计划，测试用例，测试报告等）
11. 应当把 ‘尽早和不断地测试’ 作为测试人员的座右铭
12. 回归测试的关联性一定要把引起充分的注意，修改一个错误而引起更多错误出现的现象并不少见
13. 测试应该‘小规模’开始到，逐渐转向‘大规模’
14.	不可将测试用例置之度外，排除随意性
15. 必须彻底检查每一个测试结果
16. 一定要注意测试中的错误集中发生现象，这和程序员的编程水平和习惯有很大关系
17. 对测试错误结果一定要有一个确认的过程


```
